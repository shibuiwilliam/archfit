# P3.EXP.003 — Reflection/metaprogramming density bounded

| Field | Value |
|---|---|
| Principle | P3 (Shallow Explicitness) |
| Severity | `info` |
| Evidence | `medium` |
| Stability | `experimental` |
| Applies to | Go |

## What it checks

This rule counts the number of Go source files that import the `reflect`
package. If more than 3 files import `reflect`, the rule fires.

## Why it matters

The `reflect` package bypasses Go's type system. Code that dispatches
through `reflect.Value` or `reflect.Type` hides concrete types and call
paths behind `interface{}`/`any`. Static analysis tools and agents
reading source cannot follow these paths. A few reflect imports (for
serialization, test helpers) are normal; many suggest reflection-based
dispatch at the core of the design.

## How to fix

1. Replace `reflect`-based dispatch with type switches.
2. Use code generation (`go generate`) to produce type-safe wrappers.
3. Use generics (Go 1.18+) for type-parameterized utilities.
4. Restrict `reflect` usage to serialization and test packages.

```go
// Before: reflect dispatch
func process(v any) {
    rv := reflect.ValueOf(v)
    // ...
}

// After: explicit type switch
func process(v any) {
    switch x := v.(type) {
    case string:
        processString(x)
    case int:
        processInt(x)
    }
}
```

## Suppressing this rule

