# Migration guide: 0.x → 1.0

## What changed

`schema_version` moves from `0.2.0` to `1.0.0`. This guide covers every
field change since `0.1.0`.

### Added fields (non-breaking)

| Version | Field | Type | Notes |
|---|---|---|---|
| 0.2.0 | `summary.rules_with_findings` | integer | Count of rules that produced ≥1 finding. Required. |
| 0.2.0 | `findings[].llm_suggestion` | object (optional) | Present only when `--with-llm` is used. Omitted otherwise. |
| 1.0.0 | `metrics[].name` | string | Now accepts dotted names (e.g. `verification_latency_s.lint`). No pattern constraint. |

### Removed fields

None. No fields from 0.1.0 have been removed or renamed.

### Retyped fields

None.

### Ordering guarantees (unchanged)

- `findings[]`: severity desc, rule_id asc, path asc.
- `metrics[]`: name asc.
- `scores.by_principle`: keyed by `P1`–`P7`.
- All numeric scores rounded to one decimal.

## What consumers should do

1. **Accept `schema_version: "1.0.0"`** in addition to `0.x.x`.
2. **Parse `summary.rules_with_findings`** — it's now required (was absent in 0.1.0).
3. **Ignore unknown fields gracefully** — 1.x minor bumps may add optional
   fields. Do not reject documents with extra keys.

## What 1.0 freezes

From 1.0.0 onward:

- **No field removals or renames** without a 2.0.0 major bump.
- **No retyping** (string → int, etc.) without a major bump.
- **Additive changes only** in 1.x minor bumps (new optional fields).
- **Rule IDs** in `core` and `agent-tool` packs are frozen — no renumbering.
- **Exit codes 0–5** are frozen — see `docs/exit-codes.md`.
