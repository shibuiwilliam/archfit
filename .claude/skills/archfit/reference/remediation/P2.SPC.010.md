# Remediation: P2.SPC.010 — Versioned JSON output schema missing

**Summary**: The tool lacks a `schemas/*.schema.json` with a top-level `$id`.

## Decision tree

1. **Does the tool emit structured output for programmatic consumption?**
   - No → suppress with a written rationale. A tool that only speaks to humans doesn't need a schema.
   - Yes → continue.

2. **Does the author have a preferred schema identifier domain?**
   - **Ask the user** for the `$id` base URL. Inventing one (e.g., `https://example.com/...`) creates a contract the user never agreed to.

3. **Write `schemas/output.schema.json`** with the author-supplied `$id`. Start with a minimal shape — type: object, required fields from the actual output. Don't try to capture every optional field on the first pass.

4. **Add a `schema_version` field** to the tool's emitted output. Use semver. Document the compatibility promise.

5. **Re-scan** to confirm.

## Minimal schema template

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "https://yourorg.example/schemas/output.schema.json",
  "title": "my-tool output",
  "type": "object",
  "required": ["schema_version", "findings"],
  "properties": {
    "schema_version": {"type": "string", "pattern": "^[0-9]+\\.[0-9]+\\.[0-9]+$"},
    "findings": {"type": "array"}
  }
}
```

## Anti-patterns

- Inventing a random `$id`. Ask the user.
- Skipping the `schema_version` field because "the schema versions itself". Consumers need it to pin, and parsing a schema file is expensive.
