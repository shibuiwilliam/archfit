# Remediation: P5.AGG.001 — Security-sensitive files are scattered

**Summary**: Files related to auth, secrets, migrations, or deployments are spread across 3+ top-level directories. Dangerous capabilities should be concentrated.

## Decision tree

1. **Is the scatter intentional (e.g., monorepo with per-service auth)?**
   - Yes → ask the user whether to suppress with a rationale or consolidate.
   - No → continue.

2. **Identify the target directory** for consolidation.
   - Ask the user which top-level directory should own the capability.
   - Common choices: `internal/auth/`, `src/auth/`, `pkg/auth/`.

3. **Move files** to the target directory, updating import paths.
   - This is a refactoring task — do not attempt without user confirmation.

4. **Re-scan** to confirm.

## When consolidation is wrong

In monorepos or plugin architectures, each service may legitimately own
its auth adapter. In that case, suppress the rule:

```yaml
ignore:
  - rule: P5.AGG.001
    reason: "Each service owns its auth adapter; consolidated in service boundary"
    expires: 2027-06-01
```

## Checklist

- [ ] All files for the flagged category are under ≤ 2 top-level directories.
- [ ] The consolidated location is documented in AGENTS.md or CLAUDE.md.
- [ ] Import paths / references are updated throughout the codebase.
