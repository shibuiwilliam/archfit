# Remediation: P7.MRD.001 — Exit codes not documented for a CLI

**Summary**: The repo has a CLI entrypoint (`cmd/`, `bin/`, or `exe/` directory with source files) but no `docs/exit-codes.md` (or equivalent). Recognized source extensions: `.go`, `.py`, `.ts`, `.js`, `.rs`, `.rb`, `.java`, `.kt`, `.swift`, `.php`, `.sh`.

## Decision tree

1. **Is the CLI public or shared with other systems?**
   - No (personal bootstrap script) → suppress with a rationale.
   - Yes → continue.

2. **Does the author have strong opinions about the exit-code contract?**
   - **Ask the user** before inventing exit codes. Inventing a contract is worse than having none.

3. **Write `docs/exit-codes.md`** cataloging every code the CLI emits today. Start by reading the `main.go` (or equivalent) and inventorying each `os.Exit(N)` call.

4. **Re-scan** to confirm.

## Template

```markdown
# <cli> — exit codes

| Code | Meaning |
|:---:|---|
| `0` | Success. |
| `1` | <specific failure mode — NOT "generic error"> |
| `2` | <another specific failure mode> |

## Notes

- Codes above <N> are reserved.
- These codes are part of the stability contract; changing them requires an ADR.
```

## Anti-patterns to avoid

- `1` meaning "any error". Split it into specific codes.
- Not documenting `2` for usage errors, then having the CLI emit it inconsistently.
- Writing codes that don't match what the CLI actually does. Always audit `main` first.
